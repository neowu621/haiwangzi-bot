🎯 1. 為什麼這樣設計

潛水店的資料 80% 是固定的（課程、費用、潛點、潛旅地點、新手 FAQ、預約方式），舊版卻每題都丟給 AI（附一大段 system prompt + 即時資料），造成：浪費 token、回覆慢（LINE WebView 更慢）、弱模型還常算錯日期。

新版把問題分流到「成本越低越優先」的四條路：

層級方式成本負責內容
L1 點選導引（主力）快捷按鈕 → 下鑽 → 答案卡零 token / 零 DB課程、費用、潛點、潛旅地點、新手 FAQ、預約
L1.5 即時查詢點按鈕直接打既有 API零 token（快取命中零 DB）場次名額、潛旅團況、後台價目、取消/安全政策
L2 打字問 AI輸入框 → /api/assistant耗 token選單答不到的長尾自由問答
L3 真人加 LINE 找汪汪教練真人AI 也搞不定 / 要真的預約
維護心法：能放進選單就別讓 AI 回答。固定內容進 assistant-menu.ts；會變的資料交給後台（自動即時）；只有真的無法預期的問題才留給 L2 的 AI。

📁 3. 檔案地圖

檔案職責改這裡當你要…
src/lib/assistant-menu.ts選單樹（前端純靜態常數）。所有可點選的分類、子項、答案卡文字都在這新增/修改「點選」內容、價格、課程說明、FAQ
src/components/assistant/ChatWidget.tsx浮動 widget 本體。渲染選單/答案卡/連結；fetchLive() 即時查詢；打字送 /api/assistant調整外觀、互動、或新增一種「即時」查詢
src/lib/assistant-kb.ts打字問 AI 的知識庫（system prompt 固定字串）。只在 L2 自由問答時用到調整 AI 回答口吻、補 AI 該懂的背景知識
src/app/api/assistant/route.tsL2 後端：組 prompt + 呼叫 OpenRouter（Gemini 2.5 Flash）+ 工具查場次/潛旅換模型、調 AI 行為規則（一般不用動）
即時資料不在上面任何檔案 —— 場次/潛旅/價目/政策都由後台維護，widget 直接讀，改後台即生效（見第 5 節）。
內容三分工，別放錯地方：
· 點選看到的字 → assistant-menu.ts
· AI 打字回答的知識 → assistant-kb.ts + 後台「補充資訊/個性」
· 會變的即時資料（場次/團/價/政策）→ 後台，兩個檔案都別寫死

🤿 我想學潛水（課程/費用）
   ├─ 體驗潛水（免證照）      → 答案卡 NT$2,500
   ├─ OW 開放水域考證         → 答案卡 NT$14,500
   ├─ AOW 進階考證            → 答案卡 NT$14,500
   ├─ 持證 Fun Dive           → 純文字 + 連結
   └─ 裝備租借費用            → ⟳ 即時(gear)
🐠 想知道去哪潛（潛點）
   ├─ 東北角熱門潛點          → 答案卡
   └─ 船潛路線                → 答案卡
📅 最近有場次 / 還有位子？
   ├─ 本週末的場次            → ⟳ 即時(sessions-weekend)
   └─ 近兩週的場次            → ⟳ 即時(sessions-2w)
🌴 潛水旅行（潛旅）
   ├─ 現在開放報名的團        → ⟳ 即時(tours)
   └─ 潛旅地點介紹            → 答案卡
😅 新手常見問題
   ├─ 不會游泳、會怕          → 純文字
   ├─ 耳壓會痛嗎              → 純文字
   ├─ 有鯊魚嗎/危險嗎         → 純文字
   ├─ 健康狀況能潛嗎          → 純文字
   └─ 要帶什麼/裝備           → 純文字
📋 費用 / 取消退款 / 安全須知
   ├─ 取消／退款政策          → ⟳ 即時(cancel)
   ├─ 安全須知                → ⟳ 即時(safety)
   └─ 裝備租借／日潛費用      → ⟳ 即時(gear)
📝 怎麼預約？                 → 純文字 + 連結（葉節點，無子項）

每個節點的型別（MenuNode）：分支有 children；靜態葉有 card（結構化卡）或 answer（純文字），可加 links；即時葉有 live。按鈕右側符號：›=下鑽、⟳=即時查、＋=展開答案。

⚡ 5. 即時節點對應的 API（改後台即生效）

live 值打的 API資料層後台在哪維護
sessions-weekend / sessions-2w/api/trips?from=&to=第3層 版本號快取後台場次管理（新增/改場次、有人下單自動失效）
tours/api/tours第3層 版本號快取後台潛旅管理
gear/api/site-config第2層 siteConfig 快取後台 → 系統設定 → 裝備租借價 / 日潛費用
cancel / safety/api/config第2層 siteConfig 快取後台 → 系統設定 → 取消政策 / 安全須知
好消息：這些內容要更新，完全不用改程式 —— 老闆在後台存檔，版本號自動失效，下次客戶點就是最新的。維護者只在「要新增一種新的即時查詢類型」時才需動碼（見 6C）。
降級行為：即時 API 若掛掉（例：DB 連不到），widget 會 catch 錯誤並顯示「加 LINE 問汪汪教練」，不會白畫面。靜態答案卡完全不受影響。

🛠️ 6. 維護手冊（怎麼改）

6A. 改文字 / 價格 / 課程內容（最常見）

1
開 src/lib/assistant-menu.ts，找到對應節點（用 id 或 label 搜尋）
2
改 card.price / card.bullets / answer 文字，存檔
3
bump 版本 + CHANGELOG → push（見第 9 節）
⚠️ 這裡是「行銷快照」文字。會頻繁變動的價目（裝備租借/日潛費）建議走即時節點（gear）讓老闆自己在後台改，不要寫死在卡片 bullets 裡，否則每次調價都要工程改碼＋部署。

6B. 新增一個分類 / 子項 / 答案卡

只改 assistant-menu.ts 一個檔，前端純靜態、零後端。加在 ASSISTANT_MENU 陣列（新分類）或某節點的 children（新子項）。

// 一個「答案卡」葉節點
{
  id: "course-rescue",              // 全樹唯一
  label: "救援潛水員 Rescue",       // 按鈕文字（可含 emoji）
  card: {
    title: "救援潛水員（Rescue）",
    price: "NT$16,000",
    bullets: ["學會自救與救人", "含教材/裝備", "訂金 NT$6,000"],
    note: "需先具備 AOW",           // 可省略
  },
  links: [{ label: "課程介紹", href: "/#courses" }],  // 可省略
}

// 一個「純文字」葉節點（適合 FAQ）
{ id: "faq-cold", label: "冬天會冷嗎？", answer: "有 5mm 防寒衣…" }

// 一個「分支」（帶子選單）
{ id: "xxx", label: "新分類", intro: "想看哪個？", children: [ /* 上面那種葉節點 */ ] }
連結 href 用站內相對路徑：/schedule、/pclogin、/contact、/#courses、/#spots、/#trips。

6C. 新增一種「即時」查詢節點（要動兩個檔）

1
assistant-menu.ts：在 LiveKind 型別加上新種類，例如 "reviews"；並加一個 { id, label, live:"reviews" } 節點
2
ChatWidget.tsx 的 fetchLive() 加一個分支：打你要的公開 API、把結果組成 { content, links } 回傳
鐵則：要打的 API 若讀「共享資料」，務必是已包 cached() 或 getSiteConfigRow() 的既有端點（命中零 DB），不要為了小幫手裸打 Prisma。找不到合適端點就先建一支輕量 GET（比照 /api/tours）。

6D. 停用小幫手 / 改招呼語（免改碼）

後台 → 系統設定 → AI 客服：可「停用」（widget 直接隱藏）或設「自訂招呼語」。widget 開啟時打 GET /api/assistant 讀這兩個值。

6E. 調整「打字問 AI」的回答（L2）

  • 改口吻/範圍/背景知識 → src/lib/assistant-kb.ts（system prompt）。⚠️ 保持字串固定不變才能命中 prompt cache，別插入日期/隨機值。
  • 臨時補充、活動公告 → 後台「補充資訊 / 個性語氣」欄位（免改碼，會加到 prompt 後段）。
  • 換模型 → 後台 aiBot.model 或環境變數 OPENROUTER_MODEL，皆可覆寫預設 google/gemini-2.5-flash。

🚦 8. 注意事項 / 鐵則

  • 別把即時資料寫死進選單樹。場次/團/價/政策一律走 live 節點讀後台，否則資料會過期又要改碼。
  • 別為小幫手裸打 Prisma。即時查詢重用已包版本號快取的公開端點（命中零 DB）；這是全站快取分層鐵則（見 AGENTS.md、index.html 第「資料讀取四層」）。
  • 手機/LINE WebView 優先。選單與答案卡都是純文字 + inline 樣式、無外部字體/圖，首屏零額外請求；即時節點才發 1 個 API。
  • 每層都要有真人出口。LINE 綠色鈕固定在快捷區底部，別移除。
  • id 全樹唯一（React key）。新增節點記得給不重複的 id。
  • widget 目前只掛在桌機首頁 /；手機走 /mobile proxy。這行邏輯在 ChatWidget.tsx 的 onHome。

🚀 9. 部署

改到 assistant-menu.ts / ChatWidget.tsx / assistant-kb.ts 都要走部署：bump src/lib/version.ts + 更新 CHANGELOG.md → push master → Zeabur 自動建置（約 7–9 分鐘）→ curl https://haiwangzi.xyz/api/healthz 確認 version 已是新版。只改「後台資料」則不需部署，存檔即生效。
